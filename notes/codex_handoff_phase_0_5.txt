THE FIX — INSERT parameter casts
────────────────────────────────────────────────────────────────────────
Live schema (verified via psql on pg_db, db=dev):
  util.app_run_logs(app_name text, run_id uuid, verbose_log json,
                    warn_error_logs json, inserted_at timestamptz)

`$2::uuid`/`$3::json` FAILED at runtime ("error serializing parameter"): a bare
`::uuid` makes Postgres infer the PARAMETER's type as uuid, and tokio-postgres
cannot serialize a Rust String to the uuid wire type (only to text). Fixed to
`$2::text::uuid, $3::text::json, $4::text::json` — pin the param to text, cast in
SQL. Preserves the Node "send strings, let PG coerce" behavior. Fully parameterized.

────────────────────────────────────────────────────────────────────────
RUN_ENV in the container
────────────────────────────────────────────────────────────────────────
Containerized runs MUST use RUN_ENV=staging|prod, not dev. RUN_ENV=dev makes the
run-log path RELATIVE (./<app>-log...) which lands in the root-owned /workspace
while the process runs as svc (gosu) -> Permission denied. staging/prod write to
the mounted, svc-writable /opt/run-logs/<app>/. (LOGGER still controls the filename
tag, so the file is ...-log.dev.<id>.json under /opt/run-logs even with RUN_ENV
=staging — path vs filename-tag are separate, matching Node.)
